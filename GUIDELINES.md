# 開発ルール (Development Rules)

このプロジェクトでの開発において、AI（Antigravity）は以下のルールを厳守してください。

1. **ユーザーとの会話中に不明なことがあれば、質問して理解してください**:
   - 依頼文の不明箇所や、依頼時に渡されたコンテキストの不備があれば、適度に確認や質問をしてください。
   （理由：詳細が理解されず雑に作成されても、結局使えないか、使えるよう修正する負荷が多く掛る為。）
   （AIの対応可能策：「何があればこんな依頼も可能」等、依頼内容やコンテキスト要望への提案を歓迎。）

2. **依頼内容に不可能なことがあればまずはシンプルに知らせてください**:
   - ユーザーの依頼中に不可能なことがあれば、まずはシンプルに、その旨を知らせてください。
   （理由：ユーザーが実は「必ず、その依頼に限る（代替えが無い）」とは考えていない場合があるからです。）
   （AIの対応可能策：まず「その依頼は〇〇の理由で、難しい/無理があるのですが。」と知らせた後、代替え依頼の受付「代替えの依頼はありますか？」や、代替え依頼案の提案「例えば〇〇だったら可能です。」を歓迎）

3. **実装前の日本語説明**:
   - コードを修正・追加・削除する前に、必ず「どのような修正を行うか」を日本語で端的に説明してください。
   （理由：AIとの意思疎通に不慣れな当面は依頼意図の伝わりを強く確認後に実装する方が無難な為）
   （AIの対応可能策：ここでの日本語説明は必須ですが、端的な説明で構いません。）
   - 上記の端的な説明では足りなくて、詳しい説明が必要と判断された時は、その場でユーザーが会話を求めますので、その場合は、求めに応じた詳説を分かりやすく説明してください。
    （AIの対応可能策：求めに応じた詳説の例「なぜその修正が必要？」「何が改善される？」等）

4. **明示的な承認（許可）の待機**:
   - ユーザーから「実装をお願いします」「許可します」といった明示的な同意があるまで、**一切のコード操作（ファイルの作成・修正・削除など）を行わない（＝開始しない）でください。**
   （理由：ユーザーがまだ検討の仕様やテスト中の案を、AIに先回りされて実装されることを嫌う為）
   （AIの対応可能策：このルールを尊守ください）

5. **事前のコード操作の禁止**:
   - 相談中や説明の段階で、ユーザーの許可なく勝手にコードに触れることは厳禁です。
   （理由：コード修正の練習が目的で、修正箇所や方法だけを教えて欲しいと願っている時がある為）
   （AIの対応可能策：修正方法の概要を説明後「良かったらやりましょうか？」と訊ねる提案は歓迎）

6. **ルールの継続**:
   - このルールは、今後のすべての機能追加や不具合修正において継続的に適用されます。
   （理由：AIとの意思疎通に不慣れな当面は依頼意図の伝わりを強く確認後に実装する方が無難な為）
   「Dev Container内でYOLO」の環境があるレポジトリであっても、このルールは適用されますが、
   - ユーザが判断したときだけ、設定から「Strict Mode」をオフにして、このルールを解除します。その際だけはYOLOモードで開発して頂いて構いません。
   （AIの対応可能策：既に十分に単体テストや会話を尽くし、あとは実装のみとなっている状況であれば、YOLOモード可能かどうか訊ねてみて頂くことは可能です。すると、ユーザーが「Strict Mode」オフを検討する場合があります。）

7. **GitHubへのアップロード禁止**:
   - この `GUIDELINES.md` は、ローカルの Git でのみバージョン管理を行い、GitHub などのリモートリポジトリへはアップロードしないでください。リポジトリの `.gitignore` に登録して、追跡対象から外す必要があります。
   （理由：GUIDELINES.md　は、AIとユーザ間の取り決めを記載した書類であり、GITへ公開する必要がありません。）
